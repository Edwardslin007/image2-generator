image2-generator 运行原理

一句话:浏览器输入提示词 → 本地 Python 代理转发到第三方 GPT 图片 API → 拿到图片 URL 显示出来

为什么要本地代理? 因为 API_KEY 不能暴露在前端 JS 里(谁都能看到),所以让 Python 在本地中转一下,顺便加上 Authorization: Bearer sk-... 这个鉴权头。

2输入 / 输出 是什么

📥 输入 (你给系统的)

  • Prompt:文字描述,如"赛博朋克城市"
  • Size:图片尺寸 1024x1024 / 9:16 / 16:9
  • 参考图(可选,可多张):本地图片文件,会被转成 base64 dataURL 一起发上去

📤 输出 (系统给你的)

  • 一张 PNG 图片,显示在页面上
  • 可以 Download 保存到本地
  • 可以 Copy URL 拿到原始图床链接
  • 无参考图 = 文生图 (1k);有参考图 = 图生图/编辑 (2k)

3一次"生成"按钮按下去,背后发生了什么

  1. 前端打包请求:index.html 里的 JS 把 prompt 和参考图塞进 messages,模仿 OpenAI Vision 格式 ({type:"text"} + {type:"image_url"}),POST 到 /v1/chat/completions。
  2. 本地代理serve.py 拦截请求:它发现路径在白名单里 (ALLOWED_API_PATHS),就把整个 body 原样转发到 https://api.opus5.xyz,并自动加上 Authorization 头。
  3. 远端 AIgpt-image-2 生成图片:上游模型理解你的描述(可能花十几秒),把图存到 ossdown.com 之类的图床,返回一段 chat 格式的 markdown,里面藏着 ![](https://...png)。
  4. 前端正则提取 URL:JS 用 text.match(/!\[[^\]]*\]\((https?:\/\/[^)\s]+)\)/) 把图床链接抠出来。
  5. 本地代理图片再走一次代理:不直接 img.src=远端链接,而是设成 /proxy/img?url=...。serve.py 用伪装的 User-Agent 把图片下载下来回吐给浏览器。
  6. 页面显示图片。完成 ✅
为什么图片要二次代理? 图床 (ossdown.com) 会拒绝带"奇怪 UA"的请求,而且没开 CORS,浏览器直接拉会报错。Python 端伪装成 Mozilla/5.0 就能正常下载,再把图返还给前端。

4各个文件分别在干嘛

index.html
前端页面 + 全部 JS 逻辑(输入框、参考图上传、调 API、显示图片)
serve.py
本地服务器(端口 8765),静态托管 + API 反代 + 图片代理。启动入口
worker.js
Cloudflare Worker 版本的代理(部署到云上时用,功能等同 serve.py)
probe_*.py
测试脚本,用来直连 API 探测各种调用方式是否能通
test_local.py
本地端到端测试脚本
*.png
之前测试时生成的样图,留作记录

5怎么把它跑起来

  1. 在 image2-generator/ 目录里执行:python serve.py
  2. 它会自动打开浏览器,地址 http://127.0.0.1:8765/
  3. 输入提示词(可选传几张参考图),点 Generate,等十几秒,图就出来了
本地 vs 部署:本地直接 python serve.py 即可。如果想给别人用,把 worker.js 部署到 Cloudflare Workers,然后修改 index.html 里的 API_BASE 指向 worker 域名。
