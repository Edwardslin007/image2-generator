1先看清问题：为什么直连失败

前面我们测试过，App 直接访问 api.opus5.xyz 会被防火墙(GFW)在 TLS 握手阶段拒绝。原因不是 API 本身有问题，是这个域名的"姓名"出现在 TLS 握手包里时被识别拦截了。

App 在国内
   │
   │  HTTPS 请求
   │  TLS 握手包: SNI=xxx.workers.dev
   ▼
[GFW 不拦 workers.dev]
   │
   │  顺利通过
   ▼

   │

   ▼
App 收到生成的图片
核心机制 GFW 干扰的是"特定域名"的 TLS 流量。只要换一个不在黑名单上的域名当"门面"，连接就能通。Cloudflare 的 *.workers.dev 不被针对，且 Cloudflare 在国内多个城市有商业落地节点(PoP)，能直接握手成功。

2用快递比喻一秒讲懂

把"网络请求"想象成"寄快递"，整个方案就是这样：

角色 类比 位置 负责
📱 APK 下单的你 国内 发请求要生图
🟧 Cloudflare Worker 境外的转运仓库 + 代收员 边缘节点(国内可达)，但代码在境外执行 收到 App 请求 → 加上 API Key → 转给真正的 API → 把结果带回
🟩 opus5 API 真正发货的工厂 境外 真正生成图片
关键好处 APK 只跟"代收员"打交道，永远不直接联系工厂。代收员名字是 your-app.workers.dev，这个名字 GFW 不查；至于代收员背后转给谁，GFW 看不到。

1

APK 发请求

App 调用 https://your-app.workers.dev/v1/chat/completions，请求体里只有 prompt 和参考图。没有 API Key——APK 里压根不存它。
2

请求穿过 GFW 抵达 Cloudflare

因为 SNI 是 workers.dev，GFW 不拦截。Cloudflare 在国内的边缘节点(北京/上海/广州等 PoP)接收到请求，内部走 Cloudflare 全球网络送到任意一个 Worker 实例上。
3

Worker 加 Key 后转发到 opus5

Worker 代码在境外执行(没有 GFW 限制)，从 Cloudflare 的环境变量(secret)里读出 API Key，加到 Authorization 头里，再 fetch 上游 API。
4

opus5 生成图片返回

上游花 60-90 秒生成，把图片 URL 放在 chat.completion 响应的 message.content 里返回给 Worker。
5

Worker 透传响应给 APK

Worker 几乎不做加工，把 JSON 原样回传给 APK——只加一个 Access-Control-Allow-Origin 头方便兼容浏览器场景。
6

APK 提取图片 URL 并展示

APK 解析 markdown 里的 ![image](https://ossdown.com/...)，加载图片显示。下载图床这一步——如果 ossdown.com 也被墙(实测会)，APK 还需要走 Worker 的 /proxy/img?url=... 端点中转一下。我们的 worker.js 已经写好了。

4你可能的疑问

Q1: Cloudflare Worker 在国内真的稳定能访问吗？

大体可以，但不是 100%。Cloudflare 在国内通过和百度云、京东云的合作有边缘节点(China Network)，多数运营商主网线路连接 *.workers.dev 是通的，且因为 SNI 不在黑名单，GFW 不主动干扰。

极少数运营商或地区可能解析到境外节点导致变慢/被拦，遇到这种情况，可以绑定自己的域名(api.yourdomain.com)到 Worker，配上 Cloudflare 的 CDN，稳定性更好。

Q2: APK 用户要不要装什么？

不要。这是这个方案最大的优点——所有"翻墙"工作发生在 Cloudflare 那边。用户视角看：APK 调用一个 workers.dev 的 URL，秒回数据。和访问任何普通 API 完全一样。

Q3: 钱怎么算？

  • Cloudflare Worker：免费额度每天 10 万次请求，对一个图像 App 远远够用。超出后约 $5/月可以买 1000 万次额度。
  • opus5 API：按它原本的价目正常计费(每张图大约几分钱)。Worker 只是转发，不会重复计费。

Q4: API Key 安全吗？

非常安全。Key 存在 Cloudflare 的 Secret 系统里，反编译 APK 看不到 Key(因为 APK 里只写了 Worker URL)，抓 APK 的网络包也只能看到走向 Worker 的请求(密文 + 没 Key)。

如果担心有人逆向你的 APK 拿到 Worker URL 来薅你的羊毛，可以在 Worker 里加：

  • 校验 APK 自定义的 X-App-Token 头(简单粗暴)
  • 限制单个 IP 每分钟请求次数(防滥用)
  • 检查 User-Agent 是不是你的 APK

Q5: 如果 Cloudflare 哪天挂了/被封了怎么办？

方案的脆弱点确实在 Cloudflare。备选：

  • Vercel Edge Functions(类似机制，免费额度更大)
  • Deno Deploy
  • 把同样的反代代码部署到自己的境外 VPS + 国内 CDN

架构相同，只是换"中转人"。APK 端只要把 URL 配置成可热更新的(从你的服务器拉)，就能随时切换。

55 分钟部署清单

仓库里 worker.js 已经写好了，你需要做的只是把它部署到 Cloudflare：

注册 Cloudflare 账号 (免费)
访问 cloudflare.com → Sign Up，邮箱注册即可。不用绑卡也能用 Workers 免费额度。
本机装 Wrangler 命令行工具
npm install -g wrangler
wrangler login    # 打开浏览器扫码授权
在 image2-generator 仓库目录下初始化
cd image2-generator
# 创建 wrangler.toml 配置
echo 'name = "image2-proxy"
main = "worker.js"
compatibility_date = "2024-01-01"' > wrangler.toml
把 API Key 注入到 Cloudflare 的 Secret 仓库
wrangler secret put OPENAI_API_KEY
# 提示后粘贴你的 sk-xxxxx，回车
# 这个 key 永远不会出现在代码里
注入后在 Cloudflare 后台的 Workers → 你的 worker → Settings → Variables 能看到名字但看不到值。
部署
wrangler deploy
完成后会输出一个 URL，类似 https://image2-proxy.你的子域.workers.dev，这就是 APK 要用的地址。
在 APK 里写死这个 URL，发布
不需要任何代理逻辑，HTTP 客户端直接请求即可：
POST https://image2-proxy.你的子域.workers.dev/v1/chat/completions
Content-Type: application/json

{
  "model": "gpt-image-2",
  "messages": [{
    "role": "user",
    "content": [
      { "type": "text", "text": "..." },
      { "type": "image_url", "image_url": { "url": "data:..." } }
    ]
  }]
}
建议：URL 不写死在代码里 在你自己的服务器(任意国内 IP)放一个简单的 JSON 配置文件，APK 启动时拉取里面的 workerUrl。万一以后要切换到 Vercel/自有 VPS，改配置不需要重新发版 APK。
